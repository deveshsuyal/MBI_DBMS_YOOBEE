Session 01
Overview of Database Management Principles
↗ Open PDF
What is a DBMS? Software that manages databases — controlling storage, retrieval, and security of data. Examples: MySQL, Oracle, SQL Server, PostgreSQL.
🗄️ What is a DBMS?▶
  • Software that manages databases
  • Controls data storage, retrieval and security
  • Examples: MySQL, Oracle, SQL Server
  • Supports multi-user access and concurrency
🎯 Advantages▶
  • Reduced data redundancy
  • Improved data consistency
  • Better security and access control
  • Data independence from applications
📐 Database Models▶
  • Relational — tables with rows and columns
  • Hierarchical — tree structure
  • Network — graph structure
  • NoSQL — non-relational, flexible schema
🔑 Key Concepts▶
  • Schema — structure/blueprint of the database
  • Instance — actual data at a given time
  • Data Dictionary — metadata about the database
  • ACID — Atomicity, Consistency, Isolation, Durability

Session 02
Enhanced Entity-Relationship (EER) Modeling
↗ Open PDF
EER Model extends the standard ER model with specialisation, generalisation, and aggregation to model complex real-world data more accurately.
🔼 Specialisation▶
  • Top-down — dividing an entity into subclasses
  • Subclasses inherit superclass attributes
  • Example: EMPLOYEE → MANAGER, ENGINEER
  • Defined by a differentiating attribute
🔽 Generalisation▶
  • Bottom-up — combining entities into a superclass
  • Identifies common attributes across entities
  • Example: CAR, TRUCK → VEHICLE
  • Reduces redundancy in the model
🔗 Aggregation▶
  • Treats a relationship as a higher-level entity
  • Used when a relationship has its own attributes
  • Allows relationships between relationships
📋 Constraints▶
  • Disjoint — entity belongs to only one subclass
  • Overlap — entity can be in multiple subclasses
  • Total — every superclass entity must be in a subclass
  • Partial — superclass entity may not be in any subclass

Session 03
Understanding Entities and Attributes in ER Modeling
↗ Open PDF
ER Modeling uses Entities (real-world objects), Attributes (properties), and Relationships (associations) to visually design a database structure before implementation.
📦 Types of Entities▶
  • Strong Entity — has its own primary key
  • Weak Entity — depends on a strong entity
  • Associative Entity — represents a M:N relationship
🏷️ Types of Attributes▶
  • Simple — atomic, not divisible (e.g. Age)
  • Composite — can be divided (e.g. Full Name)
  • Derived — calculated from another (e.g. Age from DOB)
  • Multi-valued — can have multiple values (e.g. Phone)
🔄 Cardinality▶
  • One-to-One (1:1) — employee has one passport
  • One-to-Many (1:N) — customer places many orders
  • Many-to-Many (M:N) — students enrol in many courses
  • Participation — total (mandatory) or partial (optional)
✏️ ER Diagram Symbols▶
  • Rectangle — Entity
  • Ellipse — Attribute
  • Diamond — Relationship
  • Double rectangle — Weak entity
  • Dashed ellipse — Derived attribute

Session 04
Database Design Phases
↗ Open PDF
Database design follows a structured lifecycle — from understanding requirements to physical implementation — each phase builds on the previous.
1️⃣ Requirements Analysis▶
  • Identify data needs of the organisation
  • Interview users and stakeholders
  • Document functional requirements
  • Determine data types and volumes
2️⃣ Conceptual Design▶
  • Create an ER or EER diagram
  • Identify entities, attributes, relationships
  • Platform-independent high-level model
  • Validated with stakeholders
3️⃣ Logical Design▶
  • Convert ER diagram to relational schema
  • Define tables, columns, primary/foreign keys
  • Apply normalisation rules
  • Still independent of specific DBMS
4️⃣ Physical Design▶
  • Implement on a specific DBMS (e.g. MySQL)
  • Define storage structures and indexes
  • Optimise for performance
  • Set access controls and security

Tutorial — Database Design and Analysis
↗ Open PDF
Applied Design Session — Hands-on practice converting real-world scenarios into ER diagrams and relational schemas through structured analysis.
🧩 Scenario Analysis▶
  • Nouns in scenario = Entities
  • Verbs in scenario = Relationships
  • Determine cardinality for each relationship
  • List attributes and identify primary keys
🔁 ER to Relational Mapping▶
  • Each strong entity → one table
  • 1:N relationships → foreign key on N side
  • M:N relationships → new junction/bridge table
  • Weak entity → includes foreign key of strong entity
✅ Design Checklist▶
  • Every table has a primary key
  • Foreign keys reference valid primary keys
  • No data duplication across tables
  • Relationship cardinality correctly represented

Session 06
Functional Dependencies & Normalization in Relational Databases
↗ Open PDF
Normalisation is the process of organising a database to reduce redundancy and improve data integrity by applying a series of normal forms based on functional dependencies.
📌 Functional Dependencies▶
  • X → Y means X determines Y
  • Full FD — Y depends on the entire composite key
  • Partial FD — Y depends on part of composite key
  • Transitive FD — X → Y → Z (indirect)
1️⃣ First Normal Form (1NF)▶
  • All attributes must be atomic
  • No repeating groups or multi-valued columns
  • Each row must be unique (primary key required)
2️⃣ Second Normal Form (2NF)▶
  • Must be in 1NF first
  • No partial dependencies on composite keys
  • Every non-key attribute depends on the whole key
  • Solved by splitting into separate tables
3️⃣ Third Normal Form (3NF)▶
  • Must be in 2NF first
  • No transitive dependencies
  • Non-key attributes depend ONLY on the primary key
  • Remove transitive attributes to a new table
NormalizationTheory
Session 07
Functional Dependencies & Normalization — Extended
↗ Open PDF
Advanced Normalisation — BCNF and beyond, handling all edge cases of data anomalies including insert, update, and delete anomalies.
🅱️ BCNF▶
  • Boyce-Codd Normal Form — stronger than 3NF
  • For every FD X → Y, X must be a superkey
  • Eliminates all anomalies from functional dependencies
  • Some valid FDs may be lost during conversion
⚠️ Data Anomalies▶
  • Insertion — can't insert without unrelated data
  • Update — same data in multiple rows → inconsistency
  • Deletion — deleting a row loses other important data
  • Normalisation prevents all three
🔍 Finding FDs in Practice▶
  • Look at sample data — what determines what?
  • Draw FD diagrams to visualise dependencies
  • Find candidate keys first
  • Apply normalisation steps one at a time
📊 Denormalization▶
  • Intentional reversal of normalisation
  • Used to improve query performance
  • Trades storage efficiency for speed
  • Common in data warehouses and reporting DBs

Session 08
SQL — Part 1
↗ Open PDF
SQL (Structured Query Language) is the standard language for interacting with relational databases — used for creating, reading, updating, and deleting data.
🏗️ DDL Commands▶
  • CREATE — creates tables, databases, views
  • ALTER — modifies existing structure
  • DROP — deletes tables or databases
  • TRUNCATE — removes all rows from a table
📝 DML Commands▶
  • SELECT — retrieve data
  • INSERT — add new rows
  • UPDATE — modify existing rows
  • DELETE — remove rows
🔎 SELECT Statement▶
  • SELECT col FROM table
  • WHERE — filter rows by condition
  • ORDER BY — sort results
  • LIMIT — restrict number of returned rows
  • DISTINCT — remove duplicate rows
🔗 Constraints▶
  • PRIMARY KEY — unique identifier per row
  • FOREIGN KEY — links to another table
  • NOT NULL — value is required
  • UNIQUE — no duplicate values allowed
  • DEFAULT — default value if none provided
SQLDDLDML
Session 09
SQL — Part 2
↗ Open PDF
Intermediate SQL — JOINs, aggregate functions, and GROUP BY allow complex data analysis and queries across multiple related tables.
🔗 SQL JOINs▶
  • INNER JOIN — rows matching in both tables
  • LEFT JOIN — all from left + matching from right
  • RIGHT JOIN — all from right + matching from left
  • FULL OUTER JOIN — all rows from both tables
📊 Aggregate Functions▶
  • COUNT() — number of rows
  • SUM() — total of a numeric column
  • AVG() — average value
  • MIN() / MAX() — lowest / highest value
📂 GROUP BY & HAVING▶
  • GROUP BY — groups rows sharing a value
  • HAVING — filters groups (like WHERE for groups)
  • Clause order: WHERE → GROUP BY → HAVING
🔄 Subqueries▶
  • Query nested inside another query
  • Can appear in SELECT, WHERE, FROM
  • Correlated subquery references outer query
  • EXISTS — checks if subquery returns any result

Session 10
SQL — Part 3
↗ Open PDF
Advanced SQL — Views, stored procedures, transactions, and indexing for building robust, efficient, and maintainable database solutions.
👁️ Views▶
  • Virtual table based on a SELECT query
  • Simplifies complex, reusable queries
  • Provides security by restricting direct table access
  • CREATE VIEW name AS SELECT ...
⚡ Indexes▶
  • Speeds up data retrieval operations
  • Clustered vs non-clustered indexes
  • Too many indexes can slow down write operations
  • Best used on frequently queried columns
🔄 Transactions▶
  • BEGIN TRANSACTION — start a transaction
  • COMMIT — save all changes permanently
  • ROLLBACK — undo all changes
  • ACID properties ensure data reliability
📋 Stored Procedures & Triggers▶
  • Stored procedure — reusable SQL code block
  • Improves performance and security
  • Trigger — auto-executes on INSERT/UPDATE/DELETE
  • Used for audit logging and data validation

Session 11
Database Application Security
↗ Open PDF
Database Security protects data from unauthorised access, corruption, or theft through authentication, access controls, encryption, and auditing mechanisms.
🔐 Access Control▶
  • GRANT — give user privileges (SELECT, INSERT etc.)
  • REVOKE — remove user privileges
  • Role-Based Access Control (RBAC)
  • Principle of least privilege
💉 SQL Injection▶
  • Attacker inserts malicious SQL into input fields
  • Can bypass login or steal/delete data
  • Prevention: parameterised queries / prepared statements
  • Always validate and sanitise user input
🔒 Encryption▶
  • At rest — protects stored data files
  • In transit — SSL/TLS for database connections
  • Hashing passwords — bcrypt, SHA-256
  • Transparent Data Encryption (TDE)
📋 Auditing & Backup▶
  • Audit trails — log who accessed/changed data
  • Regular automated backups
  • Point-in-time recovery capability
  • Disaster recovery planning and testing

📖 Key Terms Glossary
Primary Key
A column that uniquely identifies each row in a table. Cannot be NULL or duplicate.
Foreign Key
A column that references the primary key of another table, establishing a link between tables.
Normalisation
The process of organising a database to reduce redundancy and improve integrity using normal forms.
Functional Dependency
X → Y means knowing X allows you to determine Y. The basis of normalisation theory.
Entity
A real-world object stored in a database, represented as a table in a relational DB.
Attribute
A property or characteristic of an entity. Represented as a column in a table.
ACID
Atomicity, Consistency, Isolation, Durability — four properties that guarantee reliable database transactions.
SQL Injection
A security attack where malicious SQL is inserted into input fields to manipulate the database.
Index
A database object that speeds up data retrieval by maintaining a sorted reference to table data.
Transaction
A unit of work executed as a whole — either fully committed or fully rolled back.
View
A virtual table based on a SELECT query. Does not store data itself but simplifies data access.
Schema
The structure or blueprint of a database — defines tables, columns, data types, and relationships.
